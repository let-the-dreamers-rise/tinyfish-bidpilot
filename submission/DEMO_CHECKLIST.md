# Demo Checklist

## Before Recording

- Use a public, judge-accessible app URL
- Make sure `live mode ready` is visible
- Keep API keys and local env files off screen
- Use one authenticated workflow if possible
- Keep one backup run ready in case the portal is slow

## Strongest Demo Flow

1. Start on the homepage hero
2. Explain the buyer: proposal manager / capture lead
3. Scroll to the ROI section
4. Show the trust stack
5. Open the command deck for the product story
6. Go to the launch pad
7. Select a real workflow
8. Show the execution policy
9. Show Vault credential item IDs if using an authenticated run
10. Launch the TinyFish run
11. Narrate login, extraction, upload, draft-save, and approval stop
12. End on the live steps and result payload

## What Judges Must See

- TinyFish is doing real live-web work
- this is not a mock or static page
- the workflow has a real business user
- the run stops at approval instead of blindly submitting
- the output is structured and useful

## Backup Demo Flow

If the authenticated portal misbehaves, use the public BidNet run:

1. show buyer and ROI
2. explain the draft-save product wedge
3. run the public read-only audit flow
4. show live steps and result payload
5. explain that the same runtime supports authenticated draft-save workflows through TinyFish Vault
